📊 Academic Management System Architecture

🔗 4.1 Data Flow Diagram (Level 1)

graph LR User([User: Admin/Staff/Student]) -->|Login| Auth[Authentication Service] Auth -->|Token| Dashboard[Dashboard Service] Dashboard -->|Mark Attendance| AttService[Attendance Logic] Dashboard -->|Enter Marks| MarkService[Marks Analytics] AttService -->|Write| DB[(Primary SQLite/Supabase)] MarkService -->|Update| DB DB -->|Read Stats| Analytics[Reports & Notifications] Analytics -->|Alerts| User

📌 Key Components:

User Layer: Admin, Staff, and Student roles

Authentication Service: Validates credentials and issues tokens

Dashboard Service: Central hub for user interactions

Business Logic: Attendance and Marks services process data

Database: Primary storage (SQLite/Supabase)

Analytics: Generates reports and alerts

🧩 4.2 Class Diagram (Core Modules)

classDiagram class User { +int id +String name +String role +login() +updateProfile() } class Student { +String reg_no +String program +float cgpa +viewAttendance() } class Staff { +String staff_id +markAttendance() +uploadMaterials() } class Admin { +manageUsers() +publishNotice() } User <|-- Student User <|-- Staff User <|-- Admin

📌 Class Hierarchy:

User (Base Class): Contains common properties (id, name, role) and methods (login, updateProfile)

Student: Inherits from User; manages registration, program, and CGPA

Staff: Inherits from User; handles attendance marking and material uploads

Admin: Inherits from User; manages system users and publishes notices
